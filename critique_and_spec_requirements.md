> Дата: 18.06.2026  
> Источник: анализ ТЗ + фактчекинг research.md + аудит архитектуры памяти

## ЧАСТЬ 1. ЖЁСТКАЯ КРИТИКА ИДЕИ

### Провал #1 — бот генерирует отчёты, а не клиентов

Текущая воронка: пользователь получает HTML со списком проблем → видит кнопку «Написать Александру» → ...конец. Почему он должен написать незнакомому фрилансеру? В отчёте нет портфолио, нет кейсов, нет цен, нет объяснения почему именно Александр. Пользователь возьмёт список проблем, пойдёт на Upwork и наймёт первого попавшегося. **Бот не создаёт конверсию — он фильтрует тёплых лидов, которые и так бы написали.**

### Провал #2 — JS-сайты и «имитация» 4 критериев

BeautifulSoup получает HTML **до** выполнения JavaScript. На React/Vue/Next.js/Tilda/Webflow бот получит пустой `<div id="root">` и скрипты. Это большинство новых сайтов малого бизнеса. Аудит выдаст «0 по 6 критериям» не потому что сайт плохой, а потому что инструмент его не видит.

Из 10 критериев **4 в текущем стеке — имитация**:

| Критерий | Что реально проверяется | Что заявлено |
|----------|------------------------|--------------|
| Работоспособность форм | Есть ли тег `<form>` | Работает ли форма |
| Удобство навигации | Считаем теги в `<nav>` | UX-аудит |
| Адаптивность дизайна | Есть ли `meta viewport` | Тест на разных экранах |
| Качество контента | Claude читает текст | Объективная оценка |

Называть это «работоспособностью форм» — значит ставить клиенту тройку за сайт который реально работает, только потому что у тебя нет Playwright.

### Провал #3 — «повторная проверка с ручным подтверждением»

Буквальный сценарий: клиент хочет повторно проверить сайт → бот спрашивает «делали правки?» → клиент говорит «да» → **Александр должен вручную подтвердить**. Когда? Если Александр спит, на встрече, в отпуске — клиент ждёт. Это не автоматизация, это создание новой очереди ручного труда. **Прямо противоположное «освобождению времени».**

### Провал #4 — 2 минуты нереальны как гарантия

PageSpeed Insights на медленном сайте = 40–70 секунд. Плюс HTTP-краулер, Claude API, генерация HTML. Два минуты — это оптимистичный сценарий. ТЗ не описывает: что происходит если PageSpeed завис на 90 секундах? Бот зависнет? Даст неполный отчёт? Таймаут не определён.

### Провал #5 — нет системы памяти (вопреки заявленному)

В ТЗ «память» = PostgreSQL с историей запросов. Это база данных, не система памяти. У бота нет: профиля пользователя, CRM для Александра, follow-up автоматики, утреннего дайджеста, приоритизации лидов. **Бот выдал отчёт и забыл о пользователе навсегда.**

### Провал #6 — нет источника трафика

ТЗ ожидает 100–250 пользователей в месяц, но не описывает откуда они возьмутся. Лендинг? Реклама? Соцсети? Без трафика это инструмент для 5–10 человек в месяц по рекомендациям.

## ЧАСТЬ 2. ФАКТЧЕКИНГ RESEARCH.MD

Три агента проверили источники. Итог:

| Утверждение | Вердикт |
|-------------|---------|
| Protocol Digital SEO Bot существует | **ФАКТ** — подтверждён |
| Habr статья №996844 существует | **ФАКТ** — подтверждён |
| MyWebAudit $49/мес | **НЕ ПОДТВЕРЖДЕНО** — сайт цену не публикует открыто |
| MyWebAudit 80% close rate | **ФАКТ одного клиента** — не среднее по рынку |
| MyWebAudit $5000+ средний чек | **Маркетинговый тезис**, не независимое исследование |
| HubSpot средний балл 60.9/100 | **ФАКТ** — реальная опубликованная цифра |
| SEOptimer $59/мес для агентств | **ФАКТ** — тариф White Label подтверждён |
| SEOptimer «2000 агентств» | **ПРЕУВЕЛИЧЕНИЕ** — точная цифра нигде не фигурирует |
| «AI выявляет 85-90% проблем vs 60-70% вручную» | **НЕТ ИСТОЧНИКА** — маркетинговые обобщения |
| «+34% конверсии от testimonials» | **ФАКТ одного кейса** (WikiJob), не универсальный закон |
| «Telegram-ботов для аудита единицы» | **ПРЕУВЕЛИЧЕНИЕ** — конкурентов больше чем показано |

**Вывод по research.md:** основные инструменты реальны, диалоги правдоподобны. Но статистические цифры (проценты конверсии, средние чеки, точность AI) — либо маркетинговые заявления компаний, либо кейсы одного клиента, либо не имеют источника. Не используй эти цифры как аргументы в питчах.

**Что реально ценно из исследования:**
- HubSpot Grader — реальный продукт с реальной механикой (email-захват, сравнение со средним)
- SEOptimer — $59/мес виджет для агентств — это рабочая бизнес-модель
- Два формата отчёта (клиент/исполнитель) — нигде нет в Telegram, это реальный дифференциатор
- Буквенные оценки A-F — психологически понятны, это стоит оставить

## ЧАСТЬ 3. ЛУЧШИЕ НАХОДКИ — ЧТО ТОЧНО ВЗЯТЬ

**Из HubSpot Grader:** сравнение со средним по рынку. Фраза «средний сайт малого бизнеса набирает 54/100, ваш — 38» бьёт сильнее чем просто «38/100». Данные есть — сам накопишь за 50 аудитов.

**Из SEOptimer:** виджет для агентств как монетизация v2. После запуска MVP можно продать белый лейбл другим разработчикам за $30–50/мес. Это пассивный доход.

**Из MyWebAudit:** идея «квалифицированный лид с болью». Не «пришёл новый пользователь», а «пришёл пользователь, у которого сайт набрал 3/10 по мобильной». Разная температура лида — разный приоритет ответа.

**Из Habr-бота:** сравнение аудитов до/после. Пользователь исправил проблемы — перепроверил — увидел рост с 4/10 на 7/10. Это создаёт эмоциональную связь с ботом и с тобой.

## ЧАСТЬ 4. ОБЯЗАТЕЛЬНЫЕ ТРЕБОВАНИЯ К СПЕЦИФИКАЦИИ

Сгруппировано по приоритету.

---

### КРИТИЧНО (без этого не запускать)

**Req-01: Честно ограничить аудит или добавить JS-рендеринг**
Выбор: либо добавить Playwright/Puppeteer (тяжело, но честно), либо явно написать в боте «аудит работает на HTML-сайтах. Если ваш сайт на React/Vue/Next.js — результаты будут неточными». Иначе ставишь плохие оценки хорошим сайтам и теряешь репутацию.

**Req-02: Убрать ручное подтверждение повторных проверок**
Заменить на: cooldown 7 дней (после истечения — повторная проверка без ограничений). Никакого «Александр должен подтвердить».

**Req-03: Сократить критерии до реально измеримых**
MVP: 6 честных критериев.
- Скорость загрузки (PageSpeed Performance score)
- Мобильная версия (PageSpeed Mobile)
- SEO-базовые (title, meta description, h1, alt, robots.txt, sitemap)
- Безопасность (HTTPS + SSL)
- Контент и CTA (Claude — оффер, заголовок, призыв к действию, доверие)
- Скорость отклика сервера (TTFB)

Исключить из MVP: «Работоспособность форм», «Удобство навигации», «Адаптивность дизайна» — в текущем стеке это имитация, которая вредит доверию.

**Req-04: Добавить профиль Александра в CTA**
Минимум: в HTML-отчёте и в сообщении бота — 2–3 строки о том кто такой Александр: чем занимается, сколько сайтов сделал, ссылка на портфолио или кейс. Без этого CTA — холодный питч незнакомца.

**Req-05: Добавить таймаут и graceful degradation**
Если PageSpeed API не ответил за 70 секунд — продолжить аудит без его данных, предупредить пользователя. Не зависать.

---

### ВАЖНО (должно быть в v1, не в MVP)

**Req-06: CRM для Александра — команда /leads**
Список пользователей, которые: провели аудит, не написали, прошло 3–7 дней. Формат: имя, сайт, главная проблема, кнопка «Написать». Это главный инструмент конверсии.

**Req-07: Follow-up через 7 дней**
Автоматическое сообщение пользователю через 7 дней: «Неделю назад мы нашли X проблем на вашем сайте. Успели что-то исправить? Хотите проверить снова?» Отправляется один раз. Уведомляет Александра.

**Req-08: Утренний дайджест для Александра**
Одно сообщение в 9:00: сколько аудитов за вчера, сколько написали, сколько горячих лидов ждут. Не поток уведомлений — одна сводка.

**Req-09: Схема БД с конверсионными полями**
В таблицу users добавить: `wrote_to_alex BOOLEAN`, `became_client BOOLEAN`, `last_reminded TIMESTAMPTZ`. Это данные для /stats и принятия решений.

**Req-10: Статус пользователя в уведомлении владельцу**
Уведомление Александру должно включать «температуру» лида: если средний балл < 5 — это горячий лид, пометить явно. Не все уведомления равнозначны.

---

### РЕКОМЕНДУЕТСЯ (v2)

**Req-11: Сравнение со средним по рынку**
После накопления 50+ аудитов — добавить бенчмарк: «Средний сайт в нашей базе набирает X. Ваш — Y». Не надо выдумывать цифры — накопи свои.

**Req-12: Сравнение до/после**
При повторной проверке того же URL — показать динамику. «Было 4.2 → стало 6.8. Вы улучшились на 62%». Мотивирует возвращаться.

**Req-13: Виджет для других разработчиков (монетизация v2)**
После запуска MVP — упаковать в белый лейбл. Другие разработчики платят $30–50/мес за брендированную версию. Прецедент: SEOptimer зарабатывает на этом с 2000+ клиентов.

**Req-14: Источник трафика**
Бот без трафика — инструмент для 5 человек. Нужен хотя бы один канал: ссылка в шапке профиля, блок на основном сайте, пост в соцсетях. Прописать в спецификации как часть запуска.
